## Findings considered and rejected

### From 2026-06-13 run (original 5 plans)
- **Profile update before existence check** (`app.py:187`): `UPDATE` silently no-ops on missing ID; 404 is still returned correctly; no data risk; swap is purely cosmetic.
- **Voice assignment before character existence check** (`app.py:347`): same pattern, same verdict.
- **PipelineManager thread safety**: GIL protects simple attribute reads/writes; FastAPI's async event loop is single-threaded, making the check-then-act window vanishingly small.
- **`chapter_range` IndexError risk**: Pydantic `field_validator` at `models.py:32-43` rejects out-of-spec inputs before `app.py:216` is reached.
- **Python `pip` CVEs**: Affects the pip tool, not runtime code. Upgrade pip but not a code finding.
- **PostCSS XSS (npm)**: Build-tool dependency, not served to users. Low priority.
- **N+1 on `list_projects`** (originally a separate finding): Bundled into Plan 003 Step 5.

### From 2026-06-14 run (plans 006–010)
- **No API authentication (SEC-01)**: By-design for a localhost portfolio tool. CORS middleware is locked to `http://localhost:3000`. If this is ever deployed over a network, re-evaluate.
- **PipelineManager.start() race condition (CONCURRENT-01)**: Re-examined and re-rejected — `start_pipeline()` is an `async def` route with no `await` between the `if self.status == "running":` check and `self.status = "running"` assignment, so the single-threaded async event loop provides atomicity.
- **SQL f-string in `projects.py:update_profile()` (SECURITY-01)**: The whitelist validation at lines 90–95 validates every key before it appears in the f-string. The pattern is fine; the whitelist is sufficient.
- **SSE URL stale on env change (CORRECTNESS-01)**: `NEXT_PUBLIC_API_URL` is baked into the Next.js bundle at build time. Changing `.env.local` requires a rebuild/restart, not a hot reload; this is expected Next.js behavior, not a bug.
- **Duplicated VRAM lifecycle in LLMDirector + CastingDirector (ARCH-01)**: Real duplication, but refactoring to a shared base class is high ceremony for 40 lines and risks breaking the VRAM pattern. Low priority until a third LLM class appears.
- **Missing index on `lines(chapter_id, status)` (PERF-02)**: The UNIQUE constraint on `(chapter_id, line_index)` lets SQLite do an index scan narrowed to the chapter, then filter by status linearly. For typical chapter sizes (200–500 lines), this is fast enough. Add the index when chapters regularly exceed 1000 lines.
- **Async routes doing synchronous DB calls (PERF-03)**: SQLite calls are sub-millisecond for this data volume. Converting 20+ routes from async to sync adds churn with negligible real-world benefit for a local single-user tool.
- **Audio element cleanup race on chapter reset (CORRECTNESS-03)**: Requires the user to reset a chapter while the audio is playing and a file deletion completes before React reconciles. Too unlikely to plan now.
- **`_query_vram_mb` bare except (CORRECTNESS-02)**: By-design — function is a best-effort fallback. Adding a log.warning is fine to do inline without a plan.
